Servicios involucrados 7 microservicios + Kafka + Redis + Eureka

Por que es el flujo mas complejo

El checkout es una transaccion distribuida que cruza 5 microservicios de forma sincrona (Feign) y desencadena un flujo asincrono adicional via Kafka. Implementa el patron Saga con compensacion: si cualquier paso falla despues de haber modificado estado (stock descontado, carrito limpiado), la saga ejecuta pasos compensatorios para restablecer la consistencia. Ademas involucra JWT validation, rate limiting en Redis, load balancing via Eureka, pessimistic locking en PostgreSQL para evitar race conditions de stock, y el patron Strategy para notificaciones multicanal (Email + WhatsApp via Twilio).

Flujos de compensacion (Saga)

JWT invalido o expirado 401

  1. Gateway recibe request con token invalido o expirado
  2. AuthHeaderFilter llama a JwtUtil.isValid(token) y retorna false
  3. Gateway retorna 401 Unauthorized inmediatamente
  4. Request nunca llega a ningun microservicio

Rate limit superado 429

  1. IP supera el limite: 2 req/s, burst maximo 5 para /orders
  2. Redis reporta bucket vacio
  3. Gateway retorna 429 Too Many Requests
  4. Request nunca llega al Order MS

Cliente o carrito no encontrado 400

  1. OrderService llama a CustomerMS via Feign
  2. Fallback o respuesta 404 retorna Optional.empty()
  3. Service lanza OrderException("Customer does not exist")
  4. No se creo ninguna orden, no hay nada que compensar
  5. Retorna 400 Bad Request

Stock insuficiente con compensacion 400

  1. Orden creada con status=PENDING en PostgreSQL
  2. ProductMS adquiere pessimistic lock y detecta stock insuficiente
  3. ProductMS lanza excepcion que Feign propaga a OrderService
  4. Compensacion: UPDATE orders SET status=CANCELLED
  5. Retorna 400 Bad Request
  6. El carrito NO fue limpiado y permanece intacto

Pago fallido con compensacion 400

  1. Stock ya descontado, carrito ya limpiado
  2. PaymentMS falla al procesar el pago
  3. Compensacion 1: ProductClient.restockProducts() devuelve stock
  4. Compensacion 2: UPDATE orders SET status=CANCELLED
  5. Retorna 400 Bad Request
  6. El carrito fue limpiado y no se restaura (limitacion actual)

Circuit Breaker abierto Resilience4j 400

  1. Un microservicio downstream falla mas del 50% en ventana de 10 llamadas
  2. Resilience4j abre el circuit breaker y espera 30s antes de reintentar
  3. Feign invoca el Fallback que retorna Optional.empty()
  4. Service interpreta como servicio no disponible y lanza excepcion
  5. La saga aplica compensacion segun el paso donde fallo

Notas tecnicas clave

Transaccionalidad y consistencia

@Transactional en OrderService.checkout(): garantiza atomicidad de los cambios en PostgreSQL (orden + items). Sin embargo, las llamadas Feign a servicios externos no forman parte de esta transaccion local, por eso se necesita la Saga.

Pessimistic lock en Product MS: el SELECT FOR UPDATE previene que dos checkouts concurrentes lean el mismo stock disponible al mismo tiempo. Sin este lock, dos usuarios podrian comprar el ultimo item simultaneamente.

Publicacion asincrona a Kafka: el kafkaTemplate.send() es non-blocking. Payment MS no espera confirmacion de que Notification MS recibio el evento. El response al cliente llega en cuanto el pago es COMPLETED, sin esperar que el email o WhatsApp sean enviados.

Propagacion de identidad sin JWT en microservicios: el API Gateway es el unico que valida el JWT. Los microservicios reciben X-Customer-Id y X-Customer-Role como headers HTTP trusted inyectados por el Gateway. El header original del caller es eliminado (stripped) antes de propagar el request.
